## Epic Summary

Epic 0 established the FitLife Android foundation and resolved the highest-risk technical assumptions before feature implementation begins.

Completed setup and spike items:

- SETUP-001: Multi-module Gradle project and Clean Architecture scaffold
- SETUP-002: Core domain, data, UI, MVI, theme, and reusable boilerplate
- SETUP-003: Firebase Analytics and Crashlytics integration
- SETUP-004: ML Kit pose detection performance spike
- SETUP-005: Gemini API latency spike
- SETUP-006: Room, Firestore, and WorkManager offline sync spike

Sprint status showed all six Epic 0 items as `done`, while `epic-0` was still marked `in-progress`. The retrospective treats Epic 0 as functionally complete and records the tracker correction as part of closure.

## What Went Well

- The project moved from a single app module toward the intended Clean Architecture module graph without absorbing feature work into `:app`.
- Shared MVI, domain result/error contracts, data utilities, theme ownership, and Hilt setup created reusable foundations for later stories.
- Firebase observability was added without leaking Firebase SDK types into UI or domain contracts.
- ML Kit pose detection passed the required performance spike on a real physical device with a full-body framing correction.
- The Gemini spike produced a clear negative result instead of forcing AI generation into the critical path without evidence.
- The offline sync spike evolved through review into a stronger evidence path using Room, Firestore emulator configuration, WorkManager, and server timestamp-aware reconciliation.

## Challenges

- Story and tracker alignment drifted in several places: `epic-0` stayed `in-progress`, SETUP-006 retained stale unchecked task bullets, and SETUP-003 initially required a real Firebase config before implementation could continue.
- The initial namespace mismatch between `com.aml_sakr.fitnessapp` and `com.aml_sakr.fitlife` required a sprint change proposal and correction.
- Technical spikes needed real or emulator-backed evidence. ML Kit required a physical-device framing fix; sync required stronger instrumentation evidence; Gemini required live API execution.
- Parallel Gradle validation caused cache/resource contention during SETUP-004, so sequential validation became the reliable pattern.
- Some setup work accumulated review-driven fixes, especially around cancellation handling, fallback evidence, HTTP/error classification, server timestamps, and dependency wiring.

## Key Learnings

1. Spike evidence should be environment-specific and explicit. The project avoided false confidence by rejecting invalid ML Kit runs and by recording Gemini timeout failures plainly.
2. Static fallback workout templates should be primary for v1.0 plan generation; Gemini can remain an enhancement behind timeout and fallback boundaries.
3. Pose feedback can remain in v1.0 scope, with the caveat that later QA should broaden device coverage beyond the successful OPPO run.
4. Offline-first sync is viable enough to continue, but production promotion should reuse the emulator-backed harness and keep conflict handling under test.
5. Story files and sprint status need cleanup at closure, otherwise the team loses trust in the tracker even when code and reports are complete.

## Spike Decisions

### ML Kit Pose Detection

Decision: Pass.

Evidence:

- Physical device run on OPPO `CPH2737`
- Average processed FPS: 29.05
- p50 latency: 26 ms
- p95 latency: 36 ms
- Error count: 0
- Pose-detected frame ratio: 0.976

Recommendation:

Keep pose feedback in v1.0 and promote the spike harness carefully through the production session stories.

### Gemini API Latency

Decision: Fail.

Evidence:

- Model: `models/gemini-2.5-flash-lite`
- Live calls: 10
- Successful calls: 0
- Timeouts: 10
- Fallback count: 10

Recommendation:

Make `assets/fallback_workout_plans.json` the primary v1.0 plan source. Use Gemini as an enhancement with a strict timeout, structured parsing, and fallback behavior.

### Room + Firestore Offline Sync

Decision: Continue with offline-first pattern using the validated harness as the evidence path.

Evidence:

- Room sync entities and DAO logic compile.
- WorkManager-backed sync path is exercised in Android instrumentation.
- Firestore emulator gating is explicit for tests.
- Server timestamp-aware conflict handling is present.
- Partial upload failures and Firestore read failures are handled as failures.

Recommendation:

Promote the pattern cautiously in the production infrastructure story and keep emulator-backed validation as the regression safety net.
